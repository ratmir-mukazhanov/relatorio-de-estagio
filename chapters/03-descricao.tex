\section{Formação e Preparação Técnica}

O estágio teve início com um período de adaptação à \textit{stack} tecnológica da Devlop e ao ambiente de desenvolvimento utilizado na empresa. Para adquirir os conhecimentos práticos necessários à compreensão da base de código existente, foram frequentados os cursos de EF Core~\cite{efcore-course} e de \textit{SQL Server Database Design}~\cite{sqlserver-course} referidos no planeamento (\anexoref{anx:pit}, \autoref{tab:resumo-plano}). Complementarmente, foi desenvolvido um projeto prático \textit{CRUD} envolvendo \textit{backend} em .NET, \textit{frontend} em React e BD SQL Server, o que possibilitou a configuração de um ambiente de desenvolvimento funcional e a compreensão da interação entre as diferentes camadas tecnológicas.

Esta preparação inicial foi essencial para abordar com segurança o código do projeto principal.

\section{Análise da Arquitetura Existente}

Antes de qualquer implementação, procedeu-se a um estudo aprofundado da \textit{Tracking API} existente e do modelo de integração com o fornecedor ShipsGo. Foram analisados o orquestrador (\textit{ResolveProviderService}), responsável pelo encaminhamento de pedidos ao fornecedor adequado, e o \textit{ProviderCapabilitiesRegistry}, que define as capacidades suportadas por cada fornecedor (\anexoref{anx:tarefas}, \autoref{fig:tarefas-pag2} e \autoref{fig:tarefas-pag3}). Estudou-se ainda a lógica de registo de clientes, de \textit{shipments} e do sistema de \textit{logs} para faturação, bem como a documentação da API da SeaRates.

Desta análise resultou uma compreensão clara dos componentes do sistema, das diferenças estruturais entre os dois fornecedores - com destaque para o modelo \textit{stateless} da SeaRates - e dos desafios que a integração implicaria.

\section{Integração do Fornecedor SeaRates}

\begin{figure}[H]

A \textit{Tracking API} comunicava exclusivamente com o fornecedor ShipsGo~\cite{shipsgo}, sendo necessário acrescentar o suporte à SeaRates~\cite{searates} (\autoref{fig:searates-logo}) sem alterar os \textit{endpoints} já expostos a clientes externos. O trabalho de integração envolveu a expansão do \textit{ProviderCapabilitiesRegistry} com as novas capacidades, a criação de \textit{DTOs} para \textit{requests} e \textit{responses}, o desenvolvimento de um cliente HTTP para comunicação com a API externa e a implementação dos serviços de \textit{tracking} marítimo, aéreo e de transportadoras.

O novo fornecedor foi integrado no orquestrador através da expansão do \textit{ResolveProviderService}, mantendo o padrão de encaminhamento determinístico já utilizado no ShipsGo~\cite{shipsgo}. A \autoref{fig:orchestrator} ilustra o método após a integração.

\begin{figure}[H]
  \centering

private object ResolveProviderService(string normalizedProviderCode)

    return normalizedProviderCode switch
    {
        "SHIPSGO"  => _serviceProvider
            .GetRequiredService<IShipsgoTrackingService>(),
        "SEARATES" => _serviceProvider
            .GetRequiredService<ISeaRatesTrackingService>(),
        _          => throw new NotFoundException("Not found.")
    };
}
  \end{lstlisting}
  \caption{Método ResolveProviderService com o fornecedor SeaRates integrado}
  \label{fig:orchestrator}

Para garantir a qualidade da integração, foram desenvolvidos testes unitários e utilizou-se o Prism~\cite{prism} como ferramenta de \textit{mocking} para simular a API da SeaRates durante o desenvolvimento.

No final desta fase, a API passou a suportar ambos os fornecedores de forma transparente, com os serviços de \textit{tracking} operacionais e a comunicação com a API externa já validados. A arquitetura manteve-se compatível com os \textit{endpoints} em produção, ficando preparada para a adição de futuros fornecedores. Uma visão global da arquitetura do sistema é apresentada no \anexoref{anx:arquitetura} (\autoref{fig:arquitetura}).

\section{Desenvolvimento da Interface da Plataforma}

Com a camada de \textit{backend} consolidada, procedeu-se ao desenvolvimento de uma interface \textit{web} em React para interagir com a \textit{Tracking API}. Foram implementadas operações \textit{CRUD} para as principais entidades do sistema e desenvolvidas páginas dedicadas ao \textit{tracking} e ao detalhe de envios, integradas na API através de um \textit{data provider} personalizado. O sistema de permissões foi modelado com base em enumerados, garantindo controlo granular de acesso entre perfis de administrador e de cliente.

Do ponto de vista da experiência do utilizador, foram criados formulários e componentes personalizados e implementado um sistema de traduções com suporte para português, inglês e espanhol. Foram ainda desenvolvidas funcionalidades de gestão de créditos e, de forma iterativa, uma proposta de métricas e KPIs para a plataforma (\anexoref{anx:metricas}, \autoref{fig:metricas-pag1}, \autoref{fig:metricas-pag2} e \autoref{fig:metricas-pag3}), que serviu de base para os \textit{dashboards} do painel administrativo. A organização do código foi melhorando através de refatorações sucessivas dos componentes React.

A interface resultante (\anexoref{anx:solucao}) oferece cobertura funcional da maioria das entidades do sistema, totalmente integrada no \textit{backend}, com suporte multilíngue e um sistema de permissões robusto. Entre os ecrãs desenvolvidos, destacam-se o menu lateral e a gestão de utilizadores (\autoref{fig:solucao-menu}), a listagem de envios (\autoref{fig:solucao-envios}), a visualização dos movimentos de cada envio (\autoref{fig:solucao-movimentos}), a posição de um envio finalizado (\autoref{fig:solucao-finalizado}), o seguimento em tempo real (\autoref{fig:solucao-temporeal}), as estatísticas por cliente (\autoref{fig:solucao-cliente}) e os painéis financeiros (\autoref{fig:solucao-financeira}) e técnicos (\autoref{fig:solucao-tecnica}).

\section{Integrações e Testes Finais}

Na fase final do estágio, iniciou-se a preparação para a integração com o sistema interno de gestão de trânsitos, o i-Forwarder. Foi configurado um ambiente de desenvolvimento inicial em Four Js Genero~\cite{fourjs} (\anexoref{anx:4gl}, \autoref{fig:4gl-exemplo} e \autoref{fig:4gl-configurado}) e realizaram-se as primeiras reuniões de alinhamento com a equipa responsável. Contudo, a necessidade de garantir a estabilidade e a segurança da plataforma levou a uma reorientação das prioridades para as semanas finais.

O esforço de consolidação da integração com a SeaRates centrou-se na adição de exceções específicas para este fornecedor e na unificação e tradução das mensagens de movimento dos envios. Foram também resolvidos constrangimentos de paginação em pontos críticos do sistema, como as listagens de envios e de consumos.

O motor de estatísticas foi alvo de uma refatoração profunda: a remoção de agregações pré-calculadas simplificou a arquitetura e proporcionou maior granularidade dos dados, viabilizando a exportação para o Excel (\autoref{fig:solucao-financeira} e \autoref{fig:solucao-tecnica}).

Um dos principais desafios técnicos desta fase foi a renderização geográfica em tempo real (\autoref{fig:solucao-temporeal}). A componente de mapeamento fora inicialmente desenvolvida com dados de simulação (\anexoref{anx:searates}, \autoref{fig:searates-prism}), refletindo apenas rotas de envio já finalizadas.

Quando a Devlop obteve acesso à API de testes da SeaRates (\anexoref{anx:searates}, \autoref{fig:searates-teste}), a transição para dados reais - facilitada pelo fornecimento interno de contentores de teste (\anexoref{anx:searates}, \autoref{fig:searates-navios}) - expôs o problema do antimeridiano. A plataforma SeaRates fornece dados de \textit{tracking} de navios que, ao cruzarem o Oceano Pacífico, transitam de longitudes positivas (por exemplo, +179°) para longitudes negativas (por exemplo, -179°). Fisicamente, trata-se de um trajeto curto, mas, numericamente, representa um salto de 358°.

Antes da correção, os \textit{mappers} GeoJSON ligavam os pontos consecutivos através de uma abordagem linear simplista das longitudes, fazendo com que uma rota como [179°E] → [-179°W] fosse renderizada como uma linha que contornava quase todo o planeta, em vez do curto salto sobre o meridiano 180°. A solução consistiu em detetar saltos superiores a 180° entre longitudes consecutivas, normalizar as coordenadas para uma mesma reta numérica contínua, interpolar o ponto exato de cruzamento com o antimeridiano e dividir a \textit{LineString} em dois segmentos independentes, um em cada hemisfério. Como resultado, as rotas que cruzam a Linha Internacional de Data passaram a ser representadas corretamente nos mapas, exibindo a verdadeira travessia do Pacífico.

No âmbito da segurança, procedeu-se à migração do sistema de autenticação para o Keycloak~\cite{keycloak}, que passou a atuar como fornecedor de identidade único para utilizadores humanos, substituindo o modelo anterior baseado em ASP.NET Core Identity. O acesso máquina-a-máquina manteve o modelo de chaves de API, convergindo ambos os fluxos num contexto de utilizador unificado. A migração implicou a remoção dos mecanismos anteriormente utilizados para autenticação local, passando a gestão de identidade para o Keycloak. Esta alteração simplificou a arquitetura de autenticação e centralizou o controlo de acesso.

No termo desta fase, a plataforma apresentava um grau de estabilidade adequado aos requisitos identificados pela equipa no que respeita às exceções do novo fornecedor, representação geográfica correta dos trajetos marítimos, autenticação centralizada e melhorias na fiabilidade e na apresentação de dados analíticos.
